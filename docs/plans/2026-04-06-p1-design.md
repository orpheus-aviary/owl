# P1 设计文档：编辑器 + 浏览 + 回收站 + 提醒

> 日期：2026-04-06
> 状态：已确认，待实施

## 1. 设计目标

完成完整的笔记管理 GUI，解决 Go 版所有渲染和交互问题。

**核心原则：**
- 不参考 Go 版实现，从零设计（标签输入交互参考 Go 版但优化实现）
- 标签栏为唯一标签数据源，正文是纯 Markdown（`#` 永远是 Markdown 语法）
- 事件驱动提醒调度 + DB 持久化，无被动轮询
- 手动保存（Cmd+S），无自动保存，简单可靠
- 所有快捷键 P1 使用硬编码默认值，P2 设置页面支持自定义
- 同一笔记同时只能在一个 Tab 中打开（按 note ID 去重）

### 4.4 多标签（Tab）

- 编辑区顶部横向标签条
- **同一笔记单 Tab**：点击已打开的笔记 → 聚焦已有 Tab（`useEditorStore` 按 note ID 查重），不重复打开
- 支持：点击切换、`Cmd+W` 关闭、中键关闭
- 未保存标记：标签名旁显示**圆点**（正文或标签任一修改即触发）
- 关闭未保存标签时弹窗：「保存 / 不保存 / 取消」
- 标签状态通过 zustand store 管理

> P2 可考虑"笔记级 dirty 标记"（同一笔记在外部修改时提示）。

### 4.5 保存机制

**手动保存，无自动保存。**

- `Cmd+S`：保存当前笔记（内容 + 标签一起提交到 API）
- 未保存状态通过圆点标记可视化
- 关闭标签 / 关闭窗口时，未保存则弹窗提示
- 心智模型：与传统编辑器一致，未来云同步也基于手动保存触发

### 4.6 标签栏（Tag Bar）

位于编辑区下方（P2 可配置上/下），是标签数据的**唯一真相源**。

**展示：** 从 `note_tags` 表读取，渲染为 chip 列表 + 输入框。

```
[#工作 ×] [#重要 ×] [🕐 2026-04-10 15:00 ×] [⟳ 每日 ×]  [输入标签...]
```

**输入方式（Go 风格 + 优化）：**

聚焦输入框（点击或 `Cmd+L`），直接输入：

| 输入 | 行为 |
|------|------|
| `#工作` | 输入 `#` 时触发已有标签自动补全，回车确认添加 |
| `/time 2026-04-10 15:00` | 输入 `/time` 后弹出日期时间选择器辅助（也可手动输入时间），回车确认 |
| `/alarm 04-10 15:00` | 同上，弹出日期时间选择器 |
| `/daily` | 输入 `/` 时下拉提示频率修饰选项，回车确认 |

**日期时间选择器组合：**
- Calendar（shadcn）选择日期
- 额外的时间输入（HH:MM）选择时间
- Popover 方向向上展开（标签栏在底部，避免被屏幕边缘截断）

**删除：** 点击 chip 上的 `×` 按钮。

**编辑时间标签：** 点击时间类 chip → 弹出日期时间选择器修改。

**数据流：**
- 标签栏操作修改本地状态 → 触发脏标记（圆点）
- `Cmd+S` 保存时：内容 + 标签一起提交 API（`PUT /notes/:id` 带 `tags[]`）→ 写入 `note_tags` 表
- 正文不参与标签提取，`#` 在正文中完全遵循 Markdown 语法规范

**扩展性考虑：**
- 标签类型由 `TAG_TYPES` 常量定义，新增类型只需扩展该数组
- 输入框的自动补全/辅助 UI 通过 tag type 分发，新类型只需注册对应的输入辅助组件

## 8. Daemon 提醒调度器

### 8.1 架构：DB 持久化 + 事件驱动 + setTimeout 精确触发

**核心原则：DB 是提醒状态的真相源，内存 setTimeout 仅用于精确触发。**

#### 数据持久化

提醒状态持久化到 DB。在 `note_tags` 基础上增加提醒追踪：

```sql
CREATE TABLE IF NOT EXISTS reminder_status (
  note_id TEXT NOT NULL,
  tag_id TEXT NOT NULL,
  fire_at INTEGER NOT NULL,        -- 计划触发时间（Unix ms，秒和毫秒归零）
  status TEXT NOT NULL DEFAULT 'pending',  -- pending / fired
  fired_at INTEGER,                -- 实际触发时间
  PRIMARY KEY (note_id, tag_id),
  FOREIGN KEY (note_id) REFERENCES notes(id) ON DELETE CASCADE,
  FOREIGN KEY (tag_id) REFERENCES tags(id)
);
```

#### 时间精度

- `fire_at` 存储 Unix 毫秒，但**秒和毫秒始终归零**
- 用户输入 `04-10 15:00` → 解析为 `2026-04-10T15:00:00.000`
- 避免秒级误差导致"刚刚过期"的判断混乱

#### 调度流程

```
daemon 启动
  → scanOverdue(): 查询 DB 中 status='pending' AND fire_at <= now
    → 逐条发送系统通知，更新 status='fired'
  → scheduleNext(): 查询 DB 中最近的 pending 提醒
    → 设置 setTimeout 精确触发

笔记保存（涉及 /time 或 /alarm 标签变化时）
  → syncReminders(): 对比新旧标签，更新 reminder_status 表
    → 新增标签 → INSERT pending 记录
    → 删除标签 → DELETE 记录
    → 修改时间 → UPDATE fire_at + 重置 status='pending'
  → scheduleNext(): 重新计算并设置 setTimeout

setTimeout 触发
  → 发送系统通知
  → UPDATE status='fired', fired_at=now
  → 处理频率修饰（/daily → INSERT 下一条 pending 记录）
  → scheduleNext(): 设置下一条

系统休眠唤醒
  → scanOverdue(): 补检过期提醒（同启动流程）
  → scheduleNext()
```

#### 系统休眠检测（daemon 自给自足）

Daemon 是独立 Node 进程，无法使用 Electron 的 `powerMonitor`。采用轻量心跳方案：

```typescript
// 5 秒间隔检查时间跳变
private lastCheckTime = Date.now();

setInterval(() => {
  const now = Date.now();
  const delta = now - this.lastCheckTime;

  // 间隔超过 5 分钟 → 判定为从休眠恢复
  if (delta > 5 * 60 * 1000) {
    this.scanOverdue();
  }

  this.lastCheckTime = now;
}, 5000);
```

> 备选方案（不在 P1 实现）：GUI 主进程监听 `powerMonitor.on('resume')` → HTTP POST `/system/resume` → daemon scanOverdue()。但会引入 GUI-daemon 耦合。

#### 抗风险能力

| 风险场景 | 应对 |
|----------|------|
| Daemon 崩溃重启 | 启动时 scanOverdue() 补检，DB 中 pending 记录不丢失 |
| 系统休眠唤醒 | 5 秒心跳检测到时间跳变 → scanOverdue() 补发 |
| 提醒时间已过但未触发 | scanOverdue() 查询 `fire_at <= now AND status='pending'`，统一补发 |
| 重复触发 | status 字段保证幂等，fired 状态不会重复通知 |

#### 数据清理策略

- **P1**：不清理 `fired` 记录，数据量很小（一条提醒一条记录）
- **P2**：增加定期清理，保留最近 90 天 fired 记录

### 8.2 系统通知

- **GUI 在线时**：通过 Electron Notification API 发送
- **daemon 独立运行时**：通过 node-notifier 发送系统通知
- 通知内容：笔记标题 + 提醒时间

### 8.3 自动清除（回收站 Tab 2）

提醒调度器在每次 scanOverdue() 时顺带检查 Tab 2 中超期笔记（`trashed_at + 30天 < now`），永久删除。

### P1-10: reminder_status 表 + daemon 提醒调度器 + 系统通知
- 创建 `reminder_status` 表（schema + migration）

- syncReminders()：笔记保存时对比标签变化，更新 reminder_status

- scheduleNext()：查询最近 pending，设置 setTimeout
- 5 秒心跳检测系统休眠（时间跳变 > 5 分钟 → scanOverdue）
- 系统通知：Electron Notification API（GUI 在线）+ node-notifier（daemon 独立运行）
- 频率修饰处理（/daily 触发后自动创建下一条 pending）
- 回收站 Tab 2 自动清除（scanOverdue 时顺带检查）
