# P2-8 AI Chat Page — Implementation Plan

## Context

P2-7 (a → e) shipped the daemon-side AI backend: agent loop, 13 tools (read + Tier-1/2 writes), SSE endpoint, draft/preview pipeline. P2-8 builds the GUI side: a chat page that streams agent responses, surfaces drafts and direct writes, and reconciles AI changes with whatever the user is editing.

Backend stays untouched. All work is in `packages/gui/`.

## Decisions (agreed 2026-04-17)

- **Conversation organization**: tab-style, multiple chats open at once (mirrors `EditorPage`). One backend `conversation_id` per tab.
- **Draft handoff**: manual. `draft_ready` renders a `DraftReadyCard` in the chat; user clicks "open" to spawn the editor tab.
- **Tier-1 conflicts (`note_applied` on dirty tab)**: auto-merge — append AI's text to the user's *current* content, refresh save baseline from DB, fire toast. No modal. AI loop is never blocked.
- **Tier-2 conflicts (`draft_ready` for `update` action)**: modal at **save time only** (not on arrival). Two choices: `accept AI` / `keep mine`, with a `view diff` button opening a split read-only view via `@codemirror/merge`. AI loop is never blocked. Successive AI updates to the same tab overwrite the previous `pendingAiUpdate`.
- **Conflict granularity**: unified — one decision covers content + tags + folder. No per-field UI.
- **Merge UX**: no three-way merge. Binary accept/reject + side-by-side diff for visual confirmation only.
- **Conversation persistence**: in-memory in daemon (per P2-7c). Closing a tab in the GUI also `DELETE /ai/conversations/:id`. Daemon restart clears everything; GUI shows that on next open.
- **External-agent preview UI**: out of scope for P2-8. CLI in P3 will handle `POST /ai/preview/apply` directly.

## Component Slice 1 — SSE Client (`lib/sse-client.ts`)

**Goal:** Parse the daemon's `event: <name>\ndata: <json>\n\n` stream into typed `AgentEvent`s.

**Why custom (vs `EventSource`):** native `EventSource` only supports GET. Our `/ai/chat` is POST with a JSON body, so we use `fetch` + `ReadableStream` and parse the wire format ourselves.

```ts
export interface SseStreamOptions {
  url: string;
  body: unknown;
  signal?: AbortSignal;
  onEvent: (event: string, data: unknown) => void;
  onError?: (err: unknown) => void;
}
export async function streamSse(options: SseStreamOptions): Promise<void>
```

Implementation notes:
- Decode chunks with `TextDecoder`, buffer until two newlines, then split into individual events.
- Tolerate SSE comment lines (`:`) and blank padding.
- Surface `signal.aborted` as a normal completion (no error).
- Single error path: throws on non-2xx response, network errors propagate via `onError`.

Tests: golden-input parsing (multi-event, partial chunks, ignored fields), abort mid-stream.

## Component Slice 2 — `ai-store` (zustand)

**Shape:**

```ts
interface ChatMessage {
  id: string;             // local uuid; no server identity
  role: 'user' | 'assistant';
  content: string;        // accumulated text deltas
  toolCalls: ChatToolCall[];
  drafts: DraftReady[];   // draft_ready cards rendered inline
  isStreaming: boolean;   // assistant message that's still receiving deltas
}

interface ChatToolCall {
  id: string;             // server tool_call_id
  name: string;
  args: Record<string, unknown>;
  result?: unknown;       // populated when tool_result arrives
  isError?: boolean;
}

interface ChatTabState {
  id: string;             // local — used as React key; matches server conversation_id once known
  conversationId: string | null;  // null until first 'conversation_id' SSE event
  title: string;          // first user message truncated; "新对话" until then
  messages: ChatMessage[];
  pendingNoteApplieds: NoteAppliedNotice[]; // queued for the toast lane
  abortController: AbortController | null;
  isStreaming: boolean;
}

interface AiState {
  chats: ChatTabState[];
  activeChatId: string | null;

  newChat(): string;                            // returns local id
  closeChat(id: string): Promise<void>;          // also DELETEs server conversation
  setActiveChat(id: string): void;
  sendMessage(chatId: string, text: string): Promise<void>;  // wires SSE
  abortStreaming(chatId: string): void;
}
```

**Key flows:**

- `sendMessage`:
  1. Push optimistic user message + empty assistant placeholder (`isStreaming=true`).
  2. Open `AbortController`, store on tab.
  3. `streamSse({ url: '/ai/chat', body: { message, conversation_id }, ... })`.
  4. Per event, call dispatcher (next slice).
  5. On `done` or abort or error, clear `abortController` and `isStreaming`.

- `abortStreaming`: calls `controller.abort()`. SSE client exits cleanly; the assistant message is left as-is (partial), trailing toast: "已停止生成".

## Component Slice 3 — Event Dispatch

Single dispatcher inside `ai-store` consumes raw SSE events and decides what to do. Centralising it keeps the SSE client dumb and makes it testable in isolation.

| Event             | Action                                                                                                       |
|-------------------|--------------------------------------------------------------------------------------------------------------|
| `conversation_id` | Set `chat.conversationId`                                                                                    |
| `message`         | Append `content` to streaming assistant message                                                              |
| `tool_call`       | Push a new `ChatToolCall` onto current assistant message                                                     |
| `tool_result`     | Patch the matching `ChatToolCall.result` / `isError`                                                         |
| `note_applied`    | (a) Forward to `editorStore.applyNoteAppliedFromAi(...)` for in-tab merge; (b) push a toast notice           |
| `draft_ready`     | Push a `DraftReady` card onto current assistant message; **do not** auto-open the editor                     |
| `preview_ready`   | Push a card with "(external preview — open via CLI)" text. No GUI action — out of scope.                     |
| `error`           | Append a system-style error bubble; clear streaming flag                                                     |
| `done`            | Clear streaming flag                                                                                         |

## Component Slice 4 — Tier-1 Auto-Merge (`applyNoteAppliedFromAi`)

New action on `editorStore`:

```ts
applyNoteAppliedFromAi(note_id: string, latestDbContent: string, appendedText: string): void
```

Behaviour:

1. If no tab open for `note_id` → no-op (toast in `ai-store` is sufficient).
2. If tab is **not dirty** → silent overwrite: `tab.content = latestDbContent`; baselines reset to `latestDbContent`.
3. If tab is **dirty** → auto-merge: `tab.content = tab.content + '\n\n' + appendedText`; `tab.originalContent = latestDbContent` (the new save baseline reflects what the DB now has). `dirty` stays `true`.

Rationale: case 3 means the user's next `Cmd+S` will diff their merged content against the DB-with-AI-append baseline; their save still goes through correctly via the existing `updateNote` path. The user sees both their edits *and* AI's append; if they don't want it, undo (CodeMirror history is preserved because we append at end rather than re-setting the whole doc).

Edge case: AI fires a second `note_applied` while the tab is still dirty — same logic applies, append again. No coalescing needed.

## Component Slice 5 — Conflict Dialog (`ConflictDialog.tsx`)

**Trigger:** the editor's Cmd+S handler calls `editorStore.requestSaveOrConflict(noteId)` (new wrapper). If the tab has a `pendingAiUpdate` AND `detectPendingUpdateConflict(...)` reports any `true`, the wrapper sets `conflictPrompt` and returns without saving — the dialog reads that state and renders. `saveNote` itself stays unchanged so direct callers keep the existing fast-path.

**Layout:**

```
┌─────────────────────────────────────────────────────┐
│ AI 对此笔记的修改与你本地的修改不一致                          │
│                                                     │
│ 冲突项: [content] [tags] [folder]                     │
│                                                     │
│ [ 接受 AI 版本 ]  [ 保留本地 ]  [ 查看差异 ]              │
└─────────────────────────────────────────────────────┘
```

Buttons map directly to `resolveConflict(decision)`:
- **接受 AI 版本** → `resolveConflict('accept-ai')` — overwrites the tab's content/tags/folderId with `pendingAiUpdate.{content,tags,folder_id}`, clears `conflictPrompt`, then calls `saveNote` (now conflict-free).
- **保留本地** → `resolveConflict('keep-mine')` — clears `tab.pendingAiUpdate`, clears `conflictPrompt`, calls `saveNote` (normal PUT path).
- **查看差异** → opens `DiffView` modal (split view, AI version on right, current local on left, both read-only). Below the diff, the same two buttons reuse `resolveConflict`.

**Where the dialog lives:** rendered globally at the editor page level, driven by `editorStore.conflictPrompt`. Not in AI page — the user might be in the editor when they hit Cmd+S.

## Risks

1. **CodeMirror merge bundle size.** `@codemirror/merge` adds ~30-40 kB. Acceptable for an Electron app; revisit only if it shows up as a bottleneck.
2. **SSE parsing edge cases.** Multi-line `data:` payloads, comment lines, BOM. Tests cover golden inputs but real LLM streams may surprise — add fallback "skip unknown line" behaviour, log to console.
3. **AI loop racing user input.** While the user is reading a streaming reply they may type the next message. Solution: input is always enabled but `Send` button disabled while `isStreaming`. Add an `Abort` button that fires `controller.abort()`.
4. **Conflict dialog timing.** If user hits Cmd+S faster than the SSE event for `draft_ready` arrives, no conflict (state hasn't been staged yet). That's fine — they're saving their pre-AI work normally. Document but don't engineer around it.
5. **Tab close mid-stream.** Concrete order in `closeChat(id)`:
   1. If `tab.abortController` exists → `controller.abort()`.
   2. **Await** the in-flight `streamSse` promise (it resolves on abort).
   3. If `tab.conversationId != null` → `await api.deleteConversation(tab.conversationId)`.
   4. Remove the tab from state, pick a new `activeChatId`.

   Doable thanks to the `conversation_id` event arriving as the FIRST SSE event (P2-7c, see plan agent-events list), so even mid-stream we already know the id. If `conversationId` is still null (server never responded), skip step 3 — there's nothing to delete.
