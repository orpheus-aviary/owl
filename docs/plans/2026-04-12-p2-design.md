# P2 设计文档：待办 + 设置 + 文件夹 + AI + 拖拽

> 日期：2026-04-12
> 状态：已确认，待实施

## 1. 设计目标

补全剩余 GUI 页面，增加 AI 能力和文件夹组织，完善用户配置。

**核心原则：**
- GUI 修改直接写 TOML 配置文件，配置即文件，方便迁移和手动编辑
- AI 工具面简单稳定，复杂度放到服务内部
- 内部 AI 写操作通过 `POST /ai/chat` 响应里携带草稿数据，GUI 自行打开未保存 Tab（无 daemon→GUI 反向通道）
- 外部 agent 写操作走 preview → apply 两步确认
- 文件夹与搜索轻度耦合：浏览页增加文件夹筛选，管理放独立面板
- 待办页遵循 P1 "编辑器 Tab 是唯一真相源" 原则：勾选前检查是否已打开

**P2 不做（延后事项）：**
- **远程连接**：涉及 daemon 多端部署、并发写、sqlite 文件锁定等复杂问题，与 P4 migration 同步机制紧密耦合，整体留到 P4 统一设计（见第 4 节说明）
- **open_note_in_gui（daemon→GUI 反向通道）**：留到 P3 CLI 场景需要时再做

## 3. 待办页面（P2-0）

### 3.1 提取方式

实时解析：打开待办页面时，从所有非回收站笔记的 content 中正则提取 `- [ ]` 和 `- [x]`。

**正则：** `/^(\s*)- \[([ xX])\] (.*)$/gm`

**开销估算：** 1000 篇笔记 × 50 行 ≈ 30~60ms，用户无感知。

### 3.2 展示方式

按笔记分组。每个有 todo 的笔记作为折叠区，标题为笔记首行。

### 3.3 筛选

两个 Tab：
- **未完成** — 只显示含 `- [ ]` 的笔记（默认）
- **全部** — 包含已完成项，已完成项划线灰显

按笔记更新时间降序排列。

### 3.4 勾选交互（openTabs 冲突处理）

遵循 P1 "编辑器 Tab 是唯一真相源" 原则。点击 checkbox 时流程：

```
点击 checkbox
  ↓
查询前端 editorStore 是否已打开该笔记 Tab
  ↓
├─ 已打开：调用 store action 直接修改该 Tab 的 content（对应行 [ ]↔[x]）
│           → Tab 自动标记为 dirty
│           → 不调用 daemon API，等用户 Cmd+S 手动保存
│           → 待办页 UI 即时反映变化（从同一 store 读，见 3.5.1）
│
└─ 未打开：调用 PATCH /notes/:id/toggle-todo 直写 DB
           → 刷新待办列表
```

**为什么这样：** 如果直接写 DB，已打开的 Tab 的 `originalContent` 会陈旧，用户后续保存时会把勾选状态覆盖回去。通过 store 修改则保持编辑器和待办页状态一致。

### 3.4.1 首次进入待办页的数据来源合并

**问题：** 用户在编辑页改了某笔记的 todo、未保存 → 切到待办页。daemon 的 `GET /todos` 扫的是 DB 内容，不包含未保存的本地修改，列表会显示旧状态。

**方案：** 待办页数据 = daemon 结果 + dirty tab overlay。

```ts
// TodoPage 加载流程
const remoteTodos = await api.getTodos({ checked, folder_id });
const dirtyTabs = useEditorStore.getState().tabs.filter(t => t.dirty);

// 按 note_id 建索引
const merged = new Map<string, TodoGroup>();
for (const g of remoteTodos) merged.set(g.note_id, g);

// Overlay：dirty tab 的 content 用同一正则解析 todos
for (const tab of dirtyTabs) {
  const localItems = parseTodosFromContent(tab.content);  // 复用 3.1 正则
  if (localItems.length > 0) {
    merged.set(tab.noteId, {
      note_id: tab.noteId,
      note_title: extractTitle(tab.content),
      items: localItems,
      hasUnsavedChanges: true,  // UI 显示"●未保存"徽标
    });
  } else {
    // 本地把 todos 删光了
    merged.delete(tab.noteId);
  }
}
```

**订阅机制：** 待办页 `useEditorStore(state => state.tabs)` 订阅 tabs 变化，任何 Tab buffer 改动（包括点 checkbox 触发的 store action）自动触发重新合并。这让 3.4 的"点击后同步"和"首次进入合并"共用同一套逻辑。

**UI 提示：** dirty 的笔记分组行右上角显示 "●未保存" 小徽标，提示用户这份数据来自未持久化的 Tab buffer。

### 3.5 API

新增 daemon 端点：

```
GET /todos?checked=false&folder_id=<uuid>
```

参数：
- `checked` (可选) — `true`/`false`，不传则返回全部
- `folder_id` (可选) — 限定文件夹范围（含子文件夹，见第 5 节递归查询）

daemon 实现：遍历非回收站笔记 content，正则提取，返回按笔记分组的 todo 列表。

响应格式：
```json
{
  "success": true,
  "data": [
    {
      "note_id": "uuid",
      "note_title": "第一行内容",
      "items": [
        { "line": 3, "text": "买菜", "checked": false },
        { "line": 5, "text": "写报告", "checked": true }
      ]
    }
  ]
}
```

### 3.6 勾选 API（仅用于未打开场景）

```
PATCH /notes/:id/toggle-todo
Body: { "line": 3 }
```

daemon 实现：读取 content，定位到指定行，切换 `[ ]` ↔ `[x]`，保存。

**注意：** 仅在笔记未在编辑器 Tab 打开时调用。已打开时走前端 store action 路径。

### 3.7 验收标准

- [ ] 待办页面显示所有笔记中的 todo 项，按笔记分组
- [ ] "未完成"Tab 只显示含未完成项的笔记
- [ ] "全部"Tab 显示所有含 todo 的笔记，已完成项灰显划线
- [ ] 笔记**未打开**时点击 checkbox：content 同步更新（走 PATCH API）
- [ ] 笔记**已打开**时点击 checkbox：通过 store action 修改 Tab content，Tab 变 dirty，不调 API
- [ ] 已打开笔记勾选后，切换到编辑页确认 dirty 状态和内容正确，手动保存后待办页状态保持
- [ ] 在编辑页改 todo 未保存 → 切换到待办页：显示本地最新 todo 状态（不是 DB 旧状态），并显示"●未保存"徽标
- [ ] 在编辑页把笔记 todo 全删光未保存 → 切换到待办页：该笔记不出现在列表
- [ ] `just check` 零错误

### 4.1.1 【已移出 P2】远程连接 → P4

> **延后原因：**
>
> 原计划的"远程连接"栏目标是让用户配置多台 daemon 的连接信息。但经过讨论，真正的用户意图是"本地 daemon + 远程 DB 读写"——这在技术上高度棘手：
>
> - `better-sqlite3` 是本地文件 API，无法直连远程 DB
> - NFS/SMB 挂载 + SQLite 文件锁已知不可靠，容易损坏
> - 多端并发写需要冲突检测（标签、reminder、content merge）
> - FTS5 索引跨端同步需要机制保证一致性
>
> 这些问题与 **P4 migration** 的 rclone bisync 同步机制紧密耦合，应作为 P4 的一部分统一设计，而不是在 P2 留一个半成品。
>
> **P4 将决定：**
>
> 1. 远程是"切换 daemon 目标"还是"本地 daemon + 远程 DB 副本同步"
> 2. 多端并发写的冲突检测与合并策略
> 3. `owl.sync.db`（已在 CLAUDE.md 约定的迁移同步副本）如何与主 DB 协同
> 4. 设置页面远程连接栏的最终 UI/API 形态
>
> **P2 相关的代码应避免引入任何远程专有抽象**（如 `remotes` 配置段、remote daemon target 切换逻辑），保持本地 daemon 假设简单清晰，为 P4 留出干净的设计空间。

## 5. 文件夹（P2-5 ~ P2-6）

### 5.1 数据模型

DB 已有 `folders` 表：`id, name, parent_id, position, created_at, updated_at, device_id`

支持无限嵌套（adjacency list model via `parent_id`）。

### 5.1.1 递归查询（Core 层）

文件夹筛选需要查询指定文件夹及其所有后代文件夹下的笔记。使用 SQLite 的 `WITH RECURSIVE` CTE 一次性展开：

```sql
WITH RECURSIVE descendants(id) AS (
  SELECT id FROM folders WHERE id = ?
  UNION ALL
  SELECT f.id FROM folders f
    JOIN descendants d ON f.parent_id = d.id
)
SELECT * FROM notes
WHERE folder_id IN (SELECT id FROM descendants)
  AND trashed_at IS NULL
```

**Core API 扩展：** `listNotes()` / `searchNotes()` 增加 `include_descendants: boolean`（默认 `true`）参数。`false` 时退回到 `folder_id = ?` 精确匹配。

**影响范围：** 待办页 `GET /todos?folder_id=` 和浏览页 `GET /notes?folder_id=` 都复用同一个递归逻辑，在 Core 层统一实现。

### 5.2 文件夹管理面板（P2-5）

**入口：** 侧边栏增加文件夹图标按钮（在 7 个页面图标之上或之下），点击展开/收起侧边文件夹树面板。

**面板内容：**
- 树形结构展示文件夹层级
- 展开/折叠子文件夹
- 右键菜单：新建子文件夹、重命名、删除
- 拖拽排序（同级内调整 position）
- 拖拽移动笔记到文件夹（从笔记列表拖入）

**API：**

```
GET    /folders              — 获取全部文件夹树
POST   /folders              — 创建文件夹
PUT    /folders/:id          — 重命名/移动文件夹
DELETE /folders/:id          — 删除文件夹（笔记 folder_id 置 null）
PATCH  /folders/reorder      — 批量更新排序
PATCH  /notes/:id/move       — 移动笔记到文件夹
```

### 5.3 浏览页文件夹筛选（P2-6）

在浏览页现有的标签筛选旁边增加文件夹下拉筛选：

- 下拉列表展示文件夹树（缩进显示层级）
- 选择文件夹 = 搜索范围限定在该文件夹（默认含子文件夹，通过 `include_descendants=true`）
- 不选 = 全部笔记（现有行为不变）
- 与标签筛选、搜索关键词可叠加
- 前端只传 `folder_id` 和 `include_descendants`，后端递归展开（见 5.1.1）

## 6. AI 工具系统（P2-7 ~ P2-8）

### 6.1 架构概览

Daemon 内置 agent loop。GUI 和外部 agent 通过同一个 HTTP API 交互。

```
GUI / CLI / MCP
    ↓ POST /ai/chat
    ↓
Daemon Agent Loop
    ├─ 接收用户消息
    ├─ 组装系统提示 + 工具定义 + 上下文
    ├─ 调用 LLM（tool-use 模式）
    ├─ 解析工具调用 → 执行 → 返回结果
    ├─ 循环直到 LLM 返回最终回答
    └─ 返回响应（含工具调用记录）
```

### 6.2 Tool Registry

#### 读工具（内外统一，7 个）

| 工具 | 参数 | 说明 |
|------|------|------|
| `search_notes` | `query, tags?, folder_id?, include_descendants?, token_budget?, limit?, offset?, date_from?, date_to?, include_trashed?` | 统一搜索入口。内部 hybrid：FTS + tag match + recency rerank。query 为空时按 recency 返回。`token_budget` 默认 8000 字符，控制返回内容累计长度。 |
| `get_note` | `id` | 读取单篇完整笔记内容+标签 |
| `list_tags` | `search?, limit?` | 列出标签，支持搜索 |
| `list_folders` | `parent_id?` | 文件夹树结构 |
| `get_reminders` | `from?, to?, status?` | 查询提醒 |
| `get_todos` | `checked?, folder_id?, include_descendants?` | 跨笔记待办查询，默认递归子文件夹 |
| `get_capabilities` | 无 | 返回所有可用工具的描述（给外部 agent 自省用） |

#### 写工具

| 工具 | 参数 | 内部 AI 行为（`source=gui`） | 外部 agent 行为（`source=external`） |
|------|------|-------------|----------------|
| `create_note` | `content, tags?, folder_id?` | daemon 不写 DB，生成 `draft_<uuid>` 占位 ID，通过 `draft_ready` SSE 事件返回草稿数据给 GUI。GUI 打开 draft Tab（`isDraft=true`），首次 Cmd+S 时走 POST /notes 真正落盘（见 6.4.2） | 返回 preview_id + 预览内容，等待 `apply_update` |
| `update_note` | `id, content?, tags?, folder_id?` | daemon 不写 DB，通过 `draft_ready` SSE 事件返回草稿数据。GUI 按 Tab 状态分流处理（见 6.4.3），dirty Tab 冲突需用户确认 | 返回 preview_id + diff 预览，等待 `apply_update` |
| `apply_update` | `preview_id` | 不使用 | 确认写入 DB |

**关键设计：** 内部 AI 路径**不依赖 daemon→GUI 反向通道**。GUI 调用 `POST /ai/chat` 时连接建立，草稿事件通过同一条 SSE 流回传，GUI 收到后调用自己的 editorStore action 打开 Tab。整个流程仍是 GUI→daemon 的单向连接模型。

#### 专项工具（3 个）

| 工具 | 参数 | 说明 |
|------|------|------|
| `append_memo` | `content` | 追加到备忘录特殊笔记 |
| `add_todo` | `content` | 追加 `- [ ] content` 到待办特殊笔记 |
| `create_reminder` | `content, fire_at, tags?, folder_id?` | 创建带 /alarm 标签的笔记 |

### 6.3 search_notes 内部策略

```
search_notes(query, token_budget=8000) 内部流程：
  1. FTS 全文匹配 → 候选集 A
  2. 标签匹配（query 中的 # 关键词）→ 候选集 B
  3. A ∪ B 去重
  4. Rerank：recency × relevance 加权排序
  5. 按累计 content 长度截断至 token_budget
     → 优先返回短笔记（信息密度高），长笔记返回摘要（首 200 字 + "..."）
```

token_budget 有默认值（`[ai].max_fts_notes` × 平均长度），调用方可覆盖。

### 6.4.1 editorStore 扩展：isDraft 字段

P1 的 `TabState` 假设 `noteId` 永远是已存在笔记的真实 ID，`saveNote` 固定走 `PUT /notes/:id`。P2 引入 AI 创建的"未落盘草稿"概念，必须扩展：

```ts
interface TabState {
  noteId: string;          // 真实 uuid 或 draft_<uuid> 占位 ID
  isDraft: boolean;        // NEW — true 表示未落盘草稿
  title: string;
  content: string;
  originalContent: string; // draft Tab 的 originalContent = "" (空白基准)
  tags: NoteTag[];
  originalTags: NoteTag[]; // draft Tab = []
  folderId: string | null; // NEW — draft Tab 首次保存时传给 POST /notes；已有笔记同步自 DB
  dirty: boolean;          // draft Tab 初始 dirty=true（相对空基准）
}
```

**注意：** P1 的现有 Tab 也需要补 `folderId` 字段（从 `GET /notes/:id` 响应同步），否则后续文件夹拖拽功能无处落地。P2-0 不涉及，等 P2-5 文件夹功能时一起补齐。

`saveNote` 分支改造。**签名保持 `Promise<boolean>` 不变**：真实 ID 只通过 `set(...)` 内部替换和事件外溢，不通过返回值外溢，避免对现有调用方（`saveActiveNote`、键盘快捷键、TabBar 关闭时的保存确认等）产生连锁改动。

```ts
saveNote: async (noteId: string): Promise<boolean> => {   // 签名不变
  const tab = get().tabs.find(t => t.noteId === noteId);
  if (!tab || !tab.dirty) return true;

  const rawTags = serializeTags(tab.tags);

  try {
    if (tab.isDraft) {
      // 首次保存：POST 创建真实笔记
      const res = await api.createNote({
        content: tab.content,
        tags: rawTags,
        folder_id: tab.folderId ?? null,
      });
      const realId = res.data.id;
      const savedTags = res.data.tags ?? tab.tags;

      // 关键：替换 tab.noteId 为真实 ID，清除 isDraft
      // 注意同步替换 activeTabId，否则 saveActiveNote 下次找不到 Tab
      set(state => ({
        tabs: state.tabs.map(t =>
          t.noteId === noteId
            ? {
                ...t,
                noteId: realId,
                isDraft: false,
                originalContent: tab.content,
                originalTags: savedTags,
                tags: savedTags,
                dirty: false,
              }
            : t,
        ),
        activeTabId: state.activeTabId === noteId ? realId : state.activeTabId,
      }));

      // 外溢通知：笔记列表刷新、URL/TabBar 更新到真实 ID
      eventBus.emit('note-created', realId);
      return true;
    } else {
      // 已有笔记：走 PUT（原有逻辑，此处略）
      const res = await api.updateNote(tab.noteId, { content: tab.content, tags: rawTags });
      // ... 原有 set() 更新 originalContent/originalTags/dirty
      return true;
    }
  } catch {
    return false;
  }
}
```

**契约总结：**
- `saveNote(noteId)` 签名保持 `Promise<boolean>`，调用方零改动
- 真实 ID 替换通过 `set()` 内部完成，`activeTabId` 同步更新
- 调用方如果需要监听 draft → 真实 ID 的转变，订阅 `eventBus.on('note-created', ...)`
- 传入的 `noteId` 参数在调用瞬间可能是 `draft_xxx`，save 返回后 store 中该 Tab 已变成真实 ID；调用方若继续持有旧 `draft_xxx` 会失效，必须通过 `useEditorStore` 重新订阅当前 activeTabId

**占位 ID 生成：** daemon 在 `create_note` 工具处理时生成 `draft_${crypto.randomUUID()}`，保证多个连续 AI create 不冲突。GUI 不验证前缀语义，只把 `isDraft` 字段当权威。

**关闭未保存 draft Tab：** 天然不调 API，用户直接丢弃草稿，DB 无变更。`closeTab` 逻辑不需要修改（已有 dirty 确认对话框）。

### 6.4.3 GUI 处理 `draft_ready`（update 分支，含 dirty 冲突处理）

```
draft_ready { action: "update", note_id, content, tags, original_content, original_tags }
  ↓
查询 editorStore 是否已有该 note_id 的 Tab
  ↓
├─ 不存在 → 打开新 Tab：
│            content = 事件.content              (AI 修改后)
│            tags = 事件.tags                    (AI 修改后)
│            originalContent = 事件.original_content  (daemon 读到的 DB 基线)
│            originalTags = 事件.original_tags       (daemon 读到的 DB 基线)
│            dirty = true（AI 修改后 ≠ 基线）
│            isDraft = false（笔记已存在于 DB）
│            切换到编辑页聚焦
│
├─ 存在且 !tab.dirty → 直接覆盖 buffer：
│            updateContent(note_id, aiContent)
│            updateTags(note_id, aiTags)
│            自动变 dirty（AI 内容 ≠ originalContent）
│            切换到编辑页聚焦
│            （安全，本地无未保存修改会丢失）
│
└─ 存在且 tab.dirty → 冲突处理：
            editorStore.setPendingAiUpdate({ noteId, content, tags })
            AI 对话页监听到 pendingAiUpdate 变化，弹出 modal：
              ┌───────────────────────────────────────┐
              │ AI 希望修改这篇笔记                    │
              │ 但你有未保存的本地修改。               │
              │                                       │
              │ [ 放弃本地修改，接受 AI 版本 ]         │
              │ [ 保留本地，忽略 AI 建议  ]           │
              │ [ 查看差异              ]             │
              └───────────────────────────────────────┘

            选 [接受 AI]    → 覆盖 buffer（同 !dirty 分支）+ 清除 pendingAiUpdate
            选 [保留本地]   → 丢弃 pendingAiUpdate，Tab 保持不变
            选 [查看差异]   → 弹 diff viewer（本地 vs AI），底部再提供二选一按钮
```

**editorStore 扩展：**

```ts
interface EditorStore {
  // ... 原有字段
  pendingAiUpdate: { noteId: string; content: string; tags: NoteTag[] } | null;
  setPendingAiUpdate: (payload: ... | null) => void;
  acceptPendingAiUpdate: () => void;  // 覆盖 buffer + 清除
  rejectPendingAiUpdate: () => void;  // 仅清除
}
```

**关键：** 用户的本地脏数据永远不会被静默覆盖。冲突 modal 是唯一入口。
