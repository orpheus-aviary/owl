## 2. 核心架构决策（P2 讨论落定）

### 2.1 CLI 与 GUI 对等独立

- CLI 通过 `npm i -g @orpheus-aviary/owl-cli` 单独安装
- GUI 通过 GitHub Releases 下载 installer
- 两者**互不拉起**
- GUI installer **打包并可选安装 CLI** 到 PATH（Docker Desktop 模式）

### 2.2 Daemon 归属

- GUI 启动时自动 spawn daemon（已有行为）
- CLI **永不自动拉 daemon**
  - GUI 在跑 → CLI 走 HTTP
  - GUI 未跑 → CLI 直连 SQLite（WAL 模式短事务）
- WAL 允许并发读 + 单写，SQLITE_BUSY 时 CLI 指数退避重试 3 次

### 2.3 没有"CLI 下载 GUI"命令

- npm 不分发 200MB 二进制
- `owl doctor` 检测环境，打印 GitHub Releases URL 供用户手动下载

### 2.4 版本命名

| 阶段完成点 | 版本 | 语义 |
|---|---|---|
| P2 完成（GUI 基础功能） | `0.2.0` | 首个有 release 的 milestone |
| P3 完成（CLI + GUI 全套） | `0.3.0` | 外部调用可用 |
| P4 完成（同步稳定） | `1.0.0` | API 契约稳定 |

用 `0.x` 起步的理由：P3 / P4 还会引入显著变动（CLI 新增公共面、P4 schema 改动），`0.x` 的 minor bump 允许 breaking，不消耗 major。

### 2.5 npm scope

- 组织名：`@orpheus-aviary`（用户需在 npm 申请免费组织）
- 命名格式：`@orpheus-aviary/<project>-<variant>`
- 示例：`@orpheus-aviary/owl-cli`、`@orpheus-aviary/owl-mcp`（可选）
- `@orpheus-aviary/owl-core` **不发布**（内部复用，外部需求不明）
- 未来 `@orpheus-aviary/lark-cli`、`@orpheus-aviary/jay-cli` 无缝扩展

### 2.6 打包不签名

- macOS：不上 Apple Developer（$99/年），用户右键 → 打开绕过 Gatekeeper
- Windows：不买 code signing 证书（$100-400/年），用户点 "仍要运行" 绕过 SmartScreen
- Linux AppImage：天然不需要签名
- 等有真实用户量抱怨再补签名

## 3. P3.1 — GUI 0.2.0 首发

### 3.1 范围

只发 GUI，不含 CLI 打包。验证 `electron-builder` + GitHub Actions 流水线。

### 3.2 技术要点

- `electron-builder.yml` 配置：`appId`、`productName`、`mac` / `win` / `linux` 三个 target
- macOS target = `dmg`（x64 + arm64 universal 或单独 arm64，看开发机）
- Windows target = `nsis`（installer）
- Linux target = `AppImage`
- 不配 `codesign` / `notarize`
- `npm run build` → `electron-builder`
- GitHub Actions workflow `.github/workflows/release-gui.yml`：
  - trigger: push tag `v0.2.*`（GUI-only tag namespace）或手动 `workflow_dispatch`
  - 矩阵：`macos-latest`、`windows-latest`、`ubuntu-latest`
  - 产物上传到 GitHub Release（`softprops/action-gh-release@v2`）

### 3.3 实施步骤

1. 安装 `electron-builder` 到 `packages/gui`
2. 配置 `packages/gui/electron-builder.yml`（单独文件，和 electron-vite 配合）
3. 本地跑通三个平台包（mac 本机能出 dmg + AppImage；win 用 wine 或交由 CI）
4. 添加 GitHub Actions release workflow
5. README 加下载说明（"macOS 右键打开"、"Windows 点仍要运行"）
6. 打 tag `v0.2.0` → CI 自动出 release

### 3.4 未决项

- **图标资源**：需要 `.icns`（macOS）、`.ico`（Windows）、`.png`（Linux）
  - 从现有 logo（`aviary/` 下）转换
- **自动更新**：P3.1 先不做，用户下载新版手动替换；P3.3 时用 `electron-updater` + GitHub Releases provider

## 4. P3.2 — CLI 核心开发

### 4.1 apps/cli 目录结构

```
apps/cli/
├── src/
│   ├── index.ts              # commander 入口
│   ├── commands/
│   │   ├── search.ts
│   │   ├── get.ts
│   │   ├── create.ts
│   │   ├── edit.ts
│   │   ├── memo.ts
│   │   ├── todo.ts
│   │   ├── remind.ts
│   │   ├── open.ts           # 打开编辑器 tab（需 reverse channel）
│   │   └── doctor.ts         # 环境诊断
│   ├── modes/
│   │   ├── http.ts           # 调 daemon 的实现
│   │   └── direct.ts         # 直连 @owl/core 的实现
│   └── lib/
│       ├── daemon-detect.ts  # HTTP /status 心跳
│       └── db-lock.ts        # SQLITE_BUSY 重试
├── package.json              # "@orpheus-aviary/owl-cli"
└── tsconfig.json
```

### 4.2 智能切换

```typescript
const daemonAlive = await detectDaemon();  // HTTP /status, 200ms timeout
const mode = daemonAlive ? 'http' : 'direct';
```

- `http` 模式：所有命令走 `fetch` 到 `http://localhost:8080`
- `direct` 模式：直接 `createDatabase()`（来自 `@owl/core`），执行完关闭
- 环境感知：`http` 模式暴露 `open` 等需要 GUI 的命令；`direct` 模式隐藏

### 4.3 打包

- `pkg` 或 `@yao-pkg/pkg` 生成单二进制（per OS）
- 但 `better-sqlite3` 是 native module，`pkg` 支持但需要手动处理 .node 二进制
- **备选**：不做单二进制，用 `npm i -g` 安装，依赖用户 Node 环境
  - 现代 CLI（Claude Code、`gh`）多数走这条路
  - 简化打包 + 更新
- **推荐备选**：优先 `npm i -g`，`pkg` 作为后续优化

### 4.4 `open_note_in_gui` 反向通道

CLI 在 http 模式想让 GUI 打开指定笔记 tab，需要 daemon → GUI 推送。

**方案对比**：

| 方案 | 复杂度 | 优点 | 缺点 |
|---|---|---|---|
| WebSocket (`@fastify/websocket`) | 中 | 全双工，未来扩展性好 | 多一个协议 |
| Server-Sent Events（已有） | 低 | AI 已用 SSE，复用基础设施 | 单向（daemon → GUI） |
| 轮询 `/events?since=<cursor>` | 低 | 无状态 | 延迟高 |

**推荐 SSE**：已有 `initSse` 基础设施，GUI 侧加一个全局事件订阅组件。路由 `/events`：
- 推送事件类型：`open_note`、未来扩展 `config_changed`、`note_applied_external` 等
- GUI mount 时订阅，收到 `open_note { id }` 调 `openNoteById(id) + navigate('/')`

### 4.5 Schema migration 系统

当前 `migrateSchema()` 是 ad-hoc `ALTER TABLE`（`notes.auto_delete_at`）。P3 前务必升级：

```
packages/core/src/db/migrations/
├── 0001_initial.sql            # 现有 schema 截图
├── 0002_add_auto_delete_at.sql
├── 0003_chat_tables.sql        # 预留给 P3.4 chat 持久化
└── ...

schema_version 表追踪已应用的 migration。
```

- 选型：`drizzle-kit` 自动生成 migrations，或手写 SQL 配合简单 runner
- **推荐手写**：schema 稳定，自动生成器的复杂度不值得
- 启动时读 `PRAGMA user_version`，依次 apply 缺失的 migration

## 5. P3.3 — 统一发 0.3.0

### 5.1 CLI 发布

```bash
# package.json bump + npm publish
cd apps/cli
npm version 0.3.0
npm publish --access public
```

### 5.2 GUI installer 打包 CLI

- `electron-builder` 的 `extraResources` 字段把 CLI 二进制（或 `npm i -g` 脚本）塞进 app bundle
- 首次启动弹窗："是否将 `owl` 命令安装到 `/usr/local/bin`？"（复刻 VS Code 的模式）
  - macOS：`sudo cp ${appBundle}/Contents/Resources/owl /usr/local/bin/`
  - Windows：installer 脚本把 `owl.exe` 加到 PATH
  - Linux：提供 `install-cli.sh`

### 5.3 GitHub Release + npm 同步

- `v0.3.0` tag 触发双路流水线：
  - `.github/workflows/release-gui.yml` → GitHub Releases 三平台包
  - `.github/workflows/release-cli.yml` → `npm publish @orpheus-aviary/owl-cli`

### 5.4 自动更新（GUI）

- 引入 `electron-updater`，provider = GitHub
- 每次启动检查 latest release，用户可选择"下载更新"
- 首次发不考虑强制更新

## 8. 风险与未决

### 8.1 已知风险

- **macOS 未签名**：Gatekeeper 弹窗可能劝退部分用户。早期可接受，有投诉再补签
- **better-sqlite3 native module**：CLI 打包单二进制会踩坑，先走 `npm i -g` 规避
- **pnpm workspace 发布**：`apps/cli` 依赖 `@owl/core`（workspace 内），npm publish 时需 `pnpm pack` 或 `pnpm publish --no-git-checks` 处理 workspace 协议。可能需要把 `@owl/core` 的必要部分拷贝到 CLI bundle，或也发布 `@owl/core`（内部范围）

### 8.2 未决项（发版前敲定）

- CLI 命令命名：`owl search` vs `owl s` / alias？
- `open_note_in_gui` SSE 通道鉴权（localhost-only 已够？）
- schema migration runner 选型（drizzle-kit vs 手写）
- 图标/Logo 资源位置
